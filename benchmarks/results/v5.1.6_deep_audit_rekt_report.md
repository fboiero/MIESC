**Date**: 2026-04-12
**Contracts**: 11 Rekt exploits with fixtures in `data/datasets/rekt_exploits/contracts/`

consensus step is gated behind `--enable-llm` for reproducibility without
requiring Ollama).

## 3. Why `properties_generated` and `defi_confirmed` read zero

The Phase 3 branches for **property generation** (access-control findings
→ auto-CVL rule) and **DeFi confirmation** (oracle/price findings → DeFi
pattern detector) fire when the finding's `type` field matches a curated
keyword set (`access|auth|owner|privileg|tx-origin|...` and
`oracle|price|manipulation|spot|flash` respectively).

On this corpus the upstream detectors (Slither, Aderyn) tag the critical
findings with their OWN vocabulary:

- Ronin (ground truth: access\_control) → detected as `reentrancy-eth`
  by Slither based on the exploit contract shape. No `access`/`auth`
  substring, so the property-generation branch does not fire.
- BonqDAO (ground truth: oracle\_manipulation) → 26 enriched findings,
  but none carried `oracle`/`price` in the `type` field.

This is a **mapping gap, not a logic bug**. The unit tests in
`tests/test_deep_audit_agent.py::TestPhase3FindingDrivenIntegration`
continue to pass because they inject findings with the expected
vocabulary. A v5.1.7 improvement would be a normalization pass that
maps Slither's `arbitrary-send-eth`, `suicidal`, `unprotected-upgrade`
taxonomy onto the Phase-3 keywords before the branches fire. That change
is intentionally deferred out of this benchmark run to keep the Bloque-3
measurement clean.

## 4. Taint analysis confirmations

`taint_confirmed` was 0 across the corpus. Inspection shows the taint
analyzer returns early on the exploit-specific harness contracts, which
typically hold one attacker function rather than the full protocol code.
Running taint on the ORIGINAL victim contracts (rather than the
attacker-side exploit fixtures) is a known gap in the corpus setup;
expanding the corpus to include victim code is queued as part of the
"Rekt victim-side contracts" work that's already on the roadmap.

---

## 5. What this benchmark validates

- Phase-3 iterations fire on real findings (not just synthetic tests):
  **80 enrichments** across 11 contracts.
- Dynamic tool triggering works as specified:
  **12 follow-up tool runs** (11× mythril on reentrancy, 1× halmos on
  oracle).
- Detection-by-class matches the v5.1.1 headline:
  **9/11 (81.8 %)** with Cohen's $\kappa = 0.77$ (unchanged).
- No regression in timing or findings-count distribution.

## 6. What it does NOT validate

- Multi-LLM consensus (`enable_llm=False` here — needs a GPU-bearing
  benchmark environment).
- Property generation and DeFi confirmation (blocked on the
  taxonomy-normalization layer documented in §3).
- Taint-confirmed counter (blocked on victim-side corpus, §4).

Those are the three concrete gates for the v5.1.7 benchmark cycle. Unlike
"add more features", each one has a specific measurable output that will
move from 0 to non-zero when unblocked.
