A stable static path is a GOOD signal here: we added capability without
regressing the well-understood baseline.

### 2.2 Comparison summary

- **Category expansion**: 8 → 13 vuln types (**+62.5 %**)
- **Types actually fired on this fixture**: 4 → 9 (**+125 %**, the 5 new
  v5.1.6 patterns all landed)
- **Real exploits covered**: zkLend (Feb 2025, $9.6 M),
  Braavos accounts (2023), Cairo 1.0 `u256` unchecked arithmetic,
  syscall-result dropping, account-abstraction signature replay.
- **Performance**: pure Python regex, ~0.01 s per contract — no change.

The "skip" rows are a latent limitation of the line-by-line scanner: the
remaining patterns need multi-line regex matching (`#[external]\n    fn ...`)
that the current `_scan` does not provide. Tracked as a follow-up — not a
v5.1.6 regression.

## 4. DeepAuditAgent Phase 3 (Bloque 3)

The agentic path gained four concrete capabilities that are NOT visible
in the static-detector competitor comparison above:

1. Oracle/price findings now actually run `DeFiPatternDetector` targeted
   at the vulnerable function.
2. Access-control findings now generate a CVL rule via `SpecGenerator`
   that pipes directly into `miesc verify --tool certora`.
3. CRITICAL findings go through a multi-LLM consensus step
   (primary + verification-use-case models) with bounded confidence
   adjustments (+0.20 / -0.30 / -0.10 depending on agreement).
4. New aggregate counters in the Phase 3 result dict:
   `needs_manual_review_count`, `properties_generated`,
   `defi_confirmed_count`.

These changes do not change the finding COUNT on the competitor harness —
they change what each CRITICAL finding CARRIES. Measuring them
quantitatively needs either (a) a labeled false-positive benchmark or
(b) time on a real incident post-mortem. Neither exists in the repo yet;
that's the v5.1.7 opportunity.

## 5. RAG knowledge base (Bloque 4)

All 60 patterns in `SWC_REGISTRY` + `EXPLOIT_EXAMPLES` carry a
`real_exploit` reference (100 % coverage). In v5.1.6 the top 8 patterns
also carry:

- `attack_steps: List[str]` — numbered exploitation recipe.
- `detection_heuristic: Optional[str]` — grep-level detection recipe.

`VulnerabilityRAG.enhance_finding()` surfaces these fields in the LLM
context block when present. Impact: LLM prompts now contain an actionable
procedure ("look for `.call{value:}`, `.transfer()`, `.send()` before a
state write; flag if no nonReentrant") rather than a prose description.

Measured impact on the static path: zero (RAG feeds the LLM, not the
static detectors). Measured impact on the LLM path: requires a labeled
benchmark that we haven't assembled yet.

## 7. What the next benchmark should measure

The 5.1.x line moved capability (LLM quality, formal verification,
agentic investigation, Cairo), but the current public harness only
measures static-detector output. To make v5.1.7's improvements visible:

1. Assemble a **labeled false-positive dataset** (auditor-validated
   findings tagged TP / FP). The `AuditorTrainedFPClassifier` shipped in
   v5.1.5 is gated on this.
2. Run the DeepAuditAgent against the Rekt exploit corpus and measure
   how many CRITICAL findings get `needs_manual_review=True` —
   a real proxy for the cost of trusting the agent's output.
3. Add a **LLM-path comparison** that compares v5.1.5 prompts
   (pre-few-shot) with v5.1.6 prompts (with few-shot + RAG `attack_steps`).
   Need a fixed seed and a consistent model.

These are follow-ups — tracked here so the next release has a shovel-ready
benchmark target rather than another round of "no regression on static
detectors".

**Conclusion.** v5.1.6 delivers a substantial capability jump
(+5 Cairo vuln types, multi-LLM consensus, real finding-driven
investigation, enriched RAG recipes, working `miesc verify`) without
regressing the static-detector baseline (97 findings, 7.2 s). The
full value of the release is LLM/agent-side and is waiting on a labeled
benchmark to measure.
